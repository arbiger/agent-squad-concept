# SOP - 仲裁與衝突處理流程

## 1. 衝突觸發條件
當執行端 Agent（如 Coder）的產出被 Reviewer 駁回，或與 SP-Expert 的原則衝突時，啟動此流程。

## 2. 三段階層仲裁機制

### 第一階段：自動修正循環
- **動作**：Main 彙整 Reviewer 與 SP-Expert（若有）的意見，回傳給執行端。
- **限制**：最多進行 2 輪循環。若 2 輪後仍未通過，進入第二階段。

### 第二階段：Main 主動裁決
- **動作**：Main 分析「執行端方案」與「Reviewer/SP-Expert 意見」的衝突點。
- **判斷基準**：
    - 是否違反 `Mission_Instruction.md` 的原始目標？
    - **SP-Expert 建議**：高權重意見，「強烈參考」，但 Main 具備終極判斷權。
    - 技術實作是否可行？
- **執行**：Main 做出最終選擇，強行指令執行端按特定方向修改。

### 第三階段：使用者介入
- **觸發條件**：
    - Main 無法判斷優劣（機率 50/50）。
    - 衝突點涉及重大設計取捨或使用者偏好。
    - SP-Expert 規範與使用者當下指令發生根本衝突。
- **動作**：Main 暫停任務，彙整爭議點並請使用者裁決。

## 3. 紀錄規範
所有仲裁過程、爭議點及最終決定，必須完整記錄於 `Mission_Log.md` 中。
